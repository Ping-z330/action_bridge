# ActionBridge 演示脚本

## 1. 项目一句话介绍

ActionBridge 是一个“会议纪要到执行闭环”的办公协作 Agent MVP。它可以把会议文本解析成摘要、决策和行动项，并通过 Web 工作台、飞书机器人、自动提醒和 Agent 调试面板，持续推进会议后的任务执行。

面试或展示时可以这样开场：

> 这个项目不是单纯的 AI 聊天 Demo，而是围绕会议后的真实执行场景做的闭环工具。它解决的是会议结束后任务没人跟、状态不透明、提醒靠人工的问题。

## 2. 演示前准备

### 启动后端

```powershell
cd E:\Ai_Project\ActionBridge\backend
.\.venv\Scripts\python.exe -m uvicorn app.main:app --reload --host 127.0.0.1 --port 8000
```

### 启动前端

```powershell
cd E:\Ai_Project\ActionBridge\frontend
npm run dev
```

### 打开页面

```text
http://localhost:3000
```

### 飞书联调检查

- `.env` 已配置 DeepSeek/OpenAI-compatible API。
- `.env` 已配置飞书 App ID、Secret、默认群会话 ID。
- 公网映射工具已把本地 `8000` 端口映射出去。
- 飞书开放平台事件订阅地址指向：

```text

```

## 3. Web 端演示流程

### 3.1 创建会议纪要

打开首页，输入会议标题和会议记录。

推荐标题：

```text
官网改版上线前同步会
```

推荐会议记录：

```text
📢 官网改版上线前准备情况同步（6月4日周四）
当前状态
新版首页视觉已基本完成
移动端适配仍存在问题，可能影响周五上线节奏

✅ 已确认决策
官网改版仍按本周五（6月5日）上线，但必须在周四下午前完成移动端验收
上线前需先通知销售和客服团队，避免客户咨询时信息不同步

📋 待办事项
负责人: 前端同学, 任务: 修复移动端导航栏错位问题, 截止时间: 周三下班前
负责人: 设计同学, 任务: 补齐首页 banner 的移动端切图, 截止时间: 周三下午前
负责人: 测试同学, 任务: 完成官网核心路径回归测试, 截止时间: 周四上午

🔜 后续跟进
产品经理：周四下午前确认上线公告文案并同步销售和客服团队
运营同学：周五上午检查官网埋点是否正常上报
```

点击“AI 生成会议纪要”。

### 3.2 展示整理结果

重点说明：

- 系统会生成会议摘要。
- 系统会提取关键决策。
- 系统会拆出行动项。
- 每个行动项都有负责人、截止时间和状态。

讲解话术：

> 这里完成了从非结构化会议文本到结构化执行数据的转换。后续所有提醒、查询和状态更新，都是围绕这些行动项展开的。

### 3.3 编辑行动项

进入会议详情页，演示：

- 修改负责人。
- 修改截止日期和截止时间。
- 修改状态为“进行中”或“已完成”。

讲解话术：

> Web 端更像后台工作台，适合负责人或项目管理员校准任务信息。飞书端则适合团队成员日常查询和更新。

### 3.4 查看任务结果页

进入：

```text
http://localhost:3000/tasks
```

展示：

- 按会议分组的任务。
- 任务完成进度。
- 风险优先筛选。
- 状态筛选和搜索。

讲解话术：

> 任务结果页解决的是“现在这个会议产生的任务推进到哪里了”。它不是单纯历史列表，而是执行看板。

### 3.5 查看历史记录页

进入：

```text
http://localhost:3000/history
```

展示：

- 会议归档。
- 整体任务数。
- 已完成、进行中、逾期、待处理统计。
- 完成率环形图。

讲解话术：

> 历史记录页更偏归档和复盘，用来回答“过去处理过哪些会议，整体执行情况如何”。

## 4. 飞书机器人演示流程

### 4.1 查看帮助

在飞书里发送：

```text
/help
```

展示机器人能力卡片。

### 4.2 从飞书创建会议

在飞书里发送：

```text
/meeting 官网改版上线前同步会

这里粘贴会议纪要正文
```

展示：

- 后端收到飞书事件。
- 创建会议和行动项。
- 机器人向群里发送会议摘要卡片。

### 4.3 查询任务

发送：

```text
/tasks
```

或自然语言：

```text
查看未完成任务
```

展示机器人返回任务卡片。

### 4.4 查询单个任务

发送：

```text
/task 2
```

展示任务详情。

### 4.5 标记任务完成

发送：

```text
/done 2
```

或自然语言：

```text
把 2 号任务标记为完成
```

展示任务状态更新。

### 4.6 修改负责人，触发确认机制

发送：

```text
把 2 号任务负责人改成测试同学
```

展示：

- Agent 识别到这是写操作。
- 机器人不会立即修改。
- 机器人发送确认提示。

然后回复：

```text
确认
```

展示负责人被更新。

讲解话术：

> 这里的重点是危险操作不会直接执行，而是进入 Pending Confirm 确认机制，避免自然语言误识别导致错误修改任务。

### 4.7 修改截止时间，触发确认机制

发送：

```text
把 3 号任务截止时间改成明天下午六点
```

然后回复：

```text
确认
```

展示截止时间更新。

### 4.8 Memory 别名演示

发送：

```text
/remember 官网 = 官网改版
```

再发送：

```text
官网进度怎么样
```

展示 Agent 能理解“官网”指的是“官网改版”。

讲解话术：

> Memory 不是向量库，而是结构化别名记忆。它适合记录团队内部简称、成员昵称和项目代号，让 Agent 更懂团队上下文。

### 4.9 项目进度总结

发送：

```text
官网改版进度怎么样
```

展示项目进度卡片，包括：

- 总任务数。
- 完成数。
- 进行中。
- 待处理。
- 风险任务。
- 总结结论。

### 4.10 主动提醒与回复闭环

手动触发后端接口：

```text
POST /api/follow-ups/run
```

或等待自动扫描。

收到提醒后，在飞书回复：

```text
完成了 #2
```

展示任务状态自动更新。

讲解话术：

> 这一步把“通知”变成了“可响应的执行闭环”。成员不用打开后台，也可以在飞书里更新状态。

## 5. Agent Debug 面板演示

打开：

```text
http://localhost:3000/agent-debug
```

### 5.1 查看 Trace

展示最近执行列表。

选择一条记录，讲解右侧链路：

- 原始输入。
- Memory 归一化。
- 意图识别。
- 工具路由。
- 执行策略。
- 最终回复。

### 5.2 手动运行 Agent

在调试输入框中输入：

```text
把第二个任务负责人改成测试同学
```

点击“运行 Agent”。

展示：

- 产生新的 Trace。
- 意图识别为负责人修改。
- 命中 `update_task_owner` 工具。
- 工具标记为危险操作。
- 需要用户确认。

讲解话术：

> 这个页面不是给普通用户用的，而是用于展示和调试 Agent 的可观测性。它证明系统不是黑盒聊天，而是有清晰的执行流、工具路由和安全策略。

## 6. 技术亮点讲法

### 6.1 LangGraph 执行流

可以这样讲：

> 我把 Agent 拆成了多个节点，包括 Memory 归一化、任务上下文加载、意图识别、任务引用解析、工具执行和响应生成。这样比把逻辑都写在一个函数里更清晰，也方便后续观察和扩展。

### 6.2 Tool Registry 工具治理

可以这样讲：

> Agent 不直接散落调用函数，而是通过 Tool Registry 管理工具。每个工具都有 source、category、dangerous、requires_confirmation 等元数据。查询类工具可以直接执行，写入类工具会走确认机制。

### 6.3 Pending Confirm 确认机制

可以这样讲：

> 对创建任务、修改负责人、修改截止时间这类危险操作，Agent 只先生成待确认动作，用户回复确认后才真正写数据库。这能降低自然语言误识别造成的风险。

### 6.4 Memory 归一化

可以这样讲：

> Memory 用来记录团队内部别名，比如“官网”等于“官网改版”。这不是为了做大而全的知识库，而是服务于任务执行中的上下文理解。

### 6.5 Agent Trace 可观测性

可以这样讲：

> 每次 Agent 执行都会留下 Trace，包括输入、归一化、意图、参数、工具、确认策略和回复。这样调试自然语言 Agent 时不会靠猜，而是能看到它每一步怎么判断。

### 6.6 飞书事件去重

可以这样讲：

> 飞书事件可能重试推送，所以我用 event_id/message_id 做数据库级幂等，避免同一条消息导致重复创建会议或重复发送卡片。

## 7. 常见面试追问回答

### 7.1 这个和飞书智能助手有什么区别？

可以回答：

> 飞书智能助手更像通用助手，而 ActionBridge 是垂直场景 Agent。它围绕会议执行闭环设计，有自己的任务数据库、确认机制、自动提醒、Trace 调试和项目进度汇总。重点不是聊天能力更强，而是把会议后的执行过程沉淀成可追踪的数据和流程。

### 7.2 为什么不直接让 LLM 修改数据库？

可以回答：

> 因为自然语言存在误识别风险。我的设计是 LLM 只负责理解和结构化意图，真正写数据库的动作必须经过工具层和确认机制。这样更安全，也更像真实业务系统。

### 7.3 为什么需要 Agent Trace？

可以回答：

> Agent 的问题通常不是代码报错，而是“它为什么这么理解”。Trace 可以把每次执行拆开看，方便排查意图识别、上下文解析和工具路由问题。

### 7.4 为什么现在没有深度接 MCP？

可以回答：

> 当前阶段先把 Tool Registry 和 Adapter 形态搭好，证明系统具备外部工具接入能力。因为项目核心是会议执行闭环，不是堆很多外部工具。后续如果要接飞书文档、日历或任务系统，可以新增适配器，而不是改 Agent 主流程。

### 7.5 项目现在算闭环了吗？

可以回答：

> 算是 MVP 闭环。它已经覆盖会议输入、AI 整理、行动项生成、任务管理、飞书通知、自然语言更新、主动提醒、回复状态更新、历史归档和 Agent 调试。后续更多是权限、部署、数据库升级和通知策略这些产品化能力。

## 8. 推荐演示顺序

如果时间只有 5 分钟：

1. Web 输入会议纪要，生成行动项。
2. 打开任务结果页，看任务分组和进度。
3. 飞书发送“查看未完成任务”。
4. 飞书发送“把 2 号任务负责人改成测试同学”，展示确认机制。
5. 打开 `/agent-debug`，展示刚才那条消息的 Trace。

如果时间有 10 分钟：

1. Web 输入会议纪要。
2. 查看详情页和任务结果页。
3. 飞书查询任务。
4. 飞书修改负责人/截止时间并确认。
5. 演示 Memory。
6. 演示项目进度总结。
7. 演示主动提醒回复。
8. 打开 Agent Debug 面板讲技术亮点。
